# Python TDD Implementation

## 基本方針

- 回答、コメント、報告は日本語で行う。
- 作業はリポジトリの `.worktree/` 配下に作成したgit worktreeで行う。
- t-wada式TDDで、Red、Green、Refactorを省略しない。
- 実装はシンプルに保ち、仕様にないフォールバックを入れない。
- 実装ブロックには、日本語の自然言語コメントを置く。
- CLI層とロジック層を分ける。
- 失敗を握りつぶさず、原因が分かるエラーとして扱う。

## TDDワークフロー

1. TODOリストを短く書く。
2. 次に実装する振る舞いを1つ選ぶ。
3. 失敗するテストを書く。
4. 期待した理由でテストが失敗することを確認する。
5. 最小限の実装でテストを通す。
6. 既存テストも含めて通す。
7. テストが通った状態でリファクタリングする。
8. TODOリストを更新する。

テストなしでロジックを追加しない。やむを得ず先に調査コードを書く場合は、最終実装前に必ずテストへ落とし込む。

## 実装スタイル

- Pythonコードは `src/<package>/` に置く。
- テストは `tests/` に置く。
- パス操作は `pathlib.Path` を使う。
- JSONの読み書きは標準ライブラリを優先する。
- 型ヒントを書く。
- 外部コマンドは `subprocess.run(..., check=True)` を基本にする。
- ファイルI/O、HTTP通信、外部コマンド実行、テキスト変換は責務を分ける。
- 将来使うかもしれない抽象化を先に作らない。
- 例外を広く捕まえて処理継続しない。

## コメント方針

実装ブロックごとに、日本語の自然言語コメントを残す。

コメントは「何をしているか」だけでなく、「なぜその処理単位にしているか」「どの仕様を守っているか」を短く説明する。

良い例:

```python
# 元のMarkdownは破壊せず、音声生成用の中間テキストとして保存する。
script_text = build_script(markdown_text)
```

避ける例:

```python
# script_textに代入する。
script_text = build_script(markdown_text)
```

1行ごとの逐語説明は避ける。複雑な処理をコメントで補うより、まず処理を小さく分ける。

## フォールバック禁止

次のようなフォールバックは入れない。

- 設定ファイルがない場合に、黙って別の場所を探す。
- 壊れたJSON辞書を空辞書として扱う。
- 入力ファイルが存在しない場合に、サンプルテキストで代替する。
- 外部コマンドが失敗した場合に、成功扱いにする。
- 外部APIに接続できない場合に、ダミーデータで処理を続ける。

必要な入力や外部依存が不足している場合は、明確に失敗させる。

## mise/uv

実装中のプロジェクトが mise で管理されている場合は、開発者向けの入口を `mise run ...` に統一する。

mise管理の判定では、リポジトリ内の `mise.toml`、`.mise.toml`、`mise.local.toml`、`.mise/config.toml`、または既存ドキュメントの `mise run ...` 記載を確認する。

mise管理プロジェクトでよく使う検証:

```bash
mise run test
mise run lint
mise run format
mise run format-check
mise run build -- content/001.md
```

miseタスクの内部では `uv run ...` を使ってよい。ユーザー向け説明では、基本的に `mise run ...` を示す。

miseで管理されていないプロジェクトでは、既存の `pyproject.toml`、`Makefile`、README、CI設定に従って検証コマンドを選ぶ。標準入口が見つからない場合だけ、`uv run pytest` や `python -m pytest` など、そのプロジェクトで自然な最小コマンドを使う。

## 完了時の報告

完了報告には次を含める。

- 実装した振る舞い
- 追加または更新したテスト
- 実行した検証コマンド
- プロジェクト標準のlinterとformatterの実行結果
- 実行できなかった検証がある場合は理由
- 変更ファイル
